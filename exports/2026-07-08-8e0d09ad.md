권한 목록 자체에서 "Read and write"를 찾는 게 아니라, **"Contents" 항목을 찾아서 그 옆의 드롭다운**에서 선택하는 방식입니다:

1. **Permissions** 섹션에서 **Repository permissions** 펼치기
2. 알파벳순 목록에서 **"Contents"** 항목 찾기 (Actions, Administration, … 사이에 있음. 설명: "Repository contents, commits, branches, downloads, releases, and merges")

이렇게 하면 됩니다:

```
Repository permissions
  ...
  Contents        Access: [Read and write ▾]   ← 이것만 변경
  ...
  Metadata        Access: [Read-only]          ← 자동으로 켜짐 (필수라 끌 수 없음, 정상)
  ...
```

- **Contents 하나만** Read and write로 바꾸면 됩니다
- **Metadata: Read-only**가 자동으로 추가되는데, 이건 fine-grained PAT의 필수 기본값이라 그대로 두면 됩니다
- 나머지는 전부 "No access" 유지

## default 브랜치 건 — 실제 이력으로 설명드릴게요

방금 재확인했는데 **지금도 default는 `feature/initial-project-setup`입니다.** main이 default라고 기억하시는 게 이상한 게 아니라, 이력을 보면 왜 이렇게 됐는지 보입니다:

| PR | 방향 | 의미 |
|----|------|------|
| #1 | → feature/initial-project-setup | 레포 첫 브랜치라 default가 됨 |
| #2 | → **main** | 이때는 main이 있었고 여기로 머지 (Swagger 작업) |
| #4 | **main →** feature/initial-project-setup | main을 setup 브랜치로 머지 — **머지 후 자동 삭제로 main이 사라짐** |
| #3 | → feature/initial-project-setup | docs 404 수정 — main이 없으니 PR 베이스가 default로 감 |

즉 **PR #4를 머지할 때 head 브랜치였던 main이 자동 삭제**됐고, 그 뒤로 원격에 main이 없는 상태였습니다 (어제 제가 워크플로우를 푸시하면서 main이 다시 생긴 것). GitHub 레포 Settings → General → Default branch에서 직접 확인 가능합니다. 그래서 docs 404 수정 같은 최근 커밋 4개가 main에는 없고, Vercel production branch는 main이라 **그 수정들은 아직 프로덕션에 배포된 적이 없어요.**

정리하려면: ① setup 브랜치 4개 커밋을 main에 머지 → ② default 브랜치를 main으로 변경 → ③ setup 브랜치 삭제. 원하시면 제가 진행합니다 (①을 푸시하면 Vercel 프로덕션 배포가 일어나 docs 수정이 반영됩니다).
