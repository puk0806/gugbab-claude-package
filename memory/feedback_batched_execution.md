브레인스토밍/설계 단계에서 개별 섹션마다 확인받지 말고, 초반 2~3개 핵심 질문으로 방향 결정 후 남은 설계는 **한 번에 제시**하고 **전체 승인 1회**만 받는다. 승인되면 곧장 구현으로 넘어간다.

**Why:** 2026-04-23~24 `@gugbab/utils` 설계 세션에서 사용자가 두 차례 "진행하라고" / "산출물을 가져오라고" 개입하며 섹션별 리뷰·스펙 문서 다듬기 단계를 중단시킴. 초기 핵심 선택(카테고리 A, deps A-1, exports A)에는 정상적으로 답했지만 그 이후 설계 리뷰·스펙 수정 루프에서 성급함을 드러냄.

**How to apply:**

- 신규 기능·패키지 시작 시 superpowers `brainstorming` 스킬은 사용하되, "섹션마다 승인" 단계는 축약한다 — 초반 방향만 대화로 정하고 남은 설계는 1메시지로 펼친 뒤 전체 승인만 받는다
- 스펙 자체 리뷰(placeholder/ambiguity check)는 내부적으로만 수행하고 사용자에게 읽어달라고 요청하지 않는다 — 사용자가 요청할 때만 제시
- 설계·계획 문서 커밋은 사용자가 명시 요청할 때만. 기본은 "코드 산출물이 메인, 문서는 보조"
- 기술 선택지 제시 시에는 2~3개 옵션 + 업계 레퍼런스(실제 `package.json` exports 조사 등) + 추천안을 함께. 추상적 분류 나열은 지양

**주의:** CLAUDE.md의 "superpowers 워크플로우로 TDD 먼저" 원칙은 유효하므로 테스트 파일은 계속 co-located로 작성. 단축하는 것은 "설계 확인 단계"이지 "테스트 작성 단계"가 아니다.
